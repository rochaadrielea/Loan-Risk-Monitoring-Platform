DAX Design Decisions

Foundation

Three Design Principles Applied to Every Measure

Every DAX measure in this model was built around three principles that ensure reliability across any dataset — not just the one used in development.

01
LN Filter in DAX

Applied inside CALCULATE — not in Power Query — so it holds regardless of what new product types appear in future datasets.

02
DIVIDE() not /

Zero-safe division that never breaks charts when the denominator is zero. Returns BLANK instead of error.

03
LASTNONBLANK for FX

Adapts to any date range automatically. No hardcoded rates. Works when fx and prices tables have different date coverage.

Measure 1

Loan Exposure

Calculates the total net loan value from LN-type transactions only. Uses the VAR pattern to eliminate floating point artifacts that appear when buys and sells cancel out exactly.

transactions table · Home measure DAX 
-- VAR pattern: calculate first, apply business logic after
Loan Exposure =
VAR Result =
    ROUND(
        CALCULATE(
            SUMX(
                transactions,
                transactions[transaction-quantity] *
                transactions[transaction-price]
            ),
            products[product-type] = "LN"
        ),
        2
    )
RETURN
    -- If net position = 0, return BLANK so Power BI hides automatically
    IF(Result = 0, BLANK(), Result)

Why This Pattern

The Problem
Floating point arithmetic in DAX produces values like 5.8E-11 when buys and sells cancel out perfectly. This is not zero — it is a binary rounding artifact. Power BI renders it as a bar or label.
Why Not ROUND
ROUND fixes the display but not the logic. A rounded 0.00 is still not BLANK — Power BI still renders the bar.
Why Not a Filter
Any hardcoded threshold like > 0.001 risks hiding real future data. A new dataset might have legitimate small positions below any chosen threshold.
The Solution
VAR + BLANK() — if the net position is exactly zero after rounding, it has no current exposure and should not appear. BLANK() is Power BI's native signal to hide. No filters needed anywhere.
Measure 2

Loan Exposure %

Expresses loan exposure as a percentage of total portfolio value. Inherits the BLANK behaviour from Loan Exposure automatically — no additional logic required.

transactions table DAX 
-- DIVIDE returns BLANK when [Loan Exposure] is BLANK
-- No additional IF() needed — cascades automatically
Loan Exposure % =
DIVIDE(
    [Loan Exposure],
    [Portfolio Value]
)
Cascade Effect

Because DIVIDE([Loan Exposure], anything) returns BLANK when the numerator is BLANK, fixing Loan Exposure once cascades automatically to Loan Exposure %. One fix at source, correct behaviour everywhere.

Measure 3

Avg LTV

Calculates average loan-to-value ratio excluding statistical outliers. Uses the same 3×IQR method as the Python pipeline — methodology is consistent across both tools.

transactions table DAX 
-- Dynamic outlier exclusion via 3×IQR fence
-- Matches exact same method used in Python pipeline
Avg LTV =
AVERAGEX(
    FILTER(
        products,
        products[product-type] = "LN" &&
        products[outlier-flag] = "OK"
    ),
    DIVIDE(1, products[loan-property-value-ratio])
)
LTV Formula
The brief provides loan-property-value-ratio which is the inverse of LTV. LTV = 1 / loan-property-value-ratio. DIVIDE() used for zero safety.
Outlier Exclusion
PR-P4N4N has a ratio of 33.33 — LTV of 0.03. Including it would pull the average to a meaningless number. The outlier-flag calculated column excludes values outside Q1−3×IQR and Q3+3×IQR dynamically.
Consistency
The 3×IQR fence is the exact same method used in the Python pipeline's IQR Outlier Detection check. Both tools flag and exclude the same records.
Measure 4

Portfolio Value

Calculates total portfolio value by combining P1-EUR and P2-USD prices at the most recent available date, with FX conversion applied to the EUR portfolio.

transactions table DAX 
-- LASTNONBLANK adapts to any date range automatically
-- No hard relationship needed between prices/fx and transactions
Portfolio Value =
VAR MaxDate = MAX(transactions[transaction-date])
VAR FXRate =
    CALCULATE(
        LASTNONBLANK(fx[EURUSD], 1),
        fx[price-date] <= MaxDate
    )
VAR P1Value =
    CALCULATE(
        LASTNONBLANK(prices[P1-EUR], 1),
        prices[price-date] <= MaxDate
    )
VAR P2Value =
    CALCULATE(
        LASTNONBLANK(prices[P2-USD], 1),
        prices[price-date] <= MaxDate
    )
RETURN
    (P1Value * FXRate) + P2Value
Why LASTNONBLANK
fx and prices tables have different date coverage than transactions. A hard relationship would break when dates don't align between dataset swaps. LASTNONBLANK looks up the most recent available value as of each transaction date — adapts automatically.
Why No Hard Join
Date ranges don't align across dataset swaps. LASTNONBLANK resolves this at calculation time — no Power Query changes needed when new data arrives.
FX Nulls
5 null EURUSD values exist in the source file. Power Query Fill Down forward-fills them before the model loads. LASTNONBLANK provides a second layer of protection at the DAX level.
